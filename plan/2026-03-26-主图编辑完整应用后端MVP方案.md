# 主图编辑完整应用后端 MVP 方案

## 1. 目标

本方案用于搭建一个**单机可跑通**的主图编辑完整应用后端 MVP。

当前阶段聚焦以下目标：

- 支撑前端的 `聊天模式` 与 `批量编辑模式`
- 将 `/home/sam/code/main_image_banana` 的聊天生图能力整合进主应用
- 将 `/home/sam/code/z_main_image` 的 PSD/PSB 工作流整合进主应用
- 建立统一的素材库、会话、任务、文件存储与本地数据库
- 保持整体实现简单、单机可运行、便于后续继续扩展

当前阶段**不考虑**：

- 通用异步任务平台
- 完整权限系统
- 多用户隔离
- 复杂部署与监控体系

## 2. 总体技术路线

### 2.1 技术栈

- 后端框架：`FastAPI`
- 数据库存储：`SQLite`
- 文件存储：本地文件系统
- 图片处理：迁入 `main_image_banana` 的 Python 能力
- PSD/PSB 处理：迁入 `z_main_image` 的 Python 能力

### 2.2 架构原则

- 保持单仓整合，不做双仓互调
- 保持单后端服务，不拆多个独立服务
- 聊天生成与 PSD 工作流都挂在同一个主后端下
- 通过模块化目录拆分不同能力域，避免后期耦合失控

## 3. 仓库结构建议

建议在当前项目内正式建立：

- `frontend/`
- `backend/`
- `plan/`

其中后端建议拆为：

- `backend/app/main.py`：应用入口
- `backend/app/config.py`：配置与目录初始化
- `backend/app/db.py`：SQLite 初始化与连接封装
- `backend/app/routers/`：路由层
- `backend/app/services/chat/`：聊天生成相关服务
- `backend/app/services/library/`：素材库相关服务
- `backend/app/services/psd/`：PSD/PSB 工作流相关服务
- `backend/app/schemas/`：Pydantic 请求/响应模型
- `backend/tests/`：后端测试

## 4. 后端能力拆分

整个主应用后端拆成 3 个核心域：

### 4.1 聊天生成域

负责聊天模式的图片编辑闭环。

职责包括：

- 创建和读取聊天会话
- 保存聊天消息
- 接收用户上传图片
- 读取上一轮摘要
- 组装 rewrite prompt
- 调用迁入后的 `main_image_banana` 能力生成结果图
- 将生成结果保存到素材库
- 将本轮结果写回聊天消息

### 4.2 素材库域

负责统一管理所有历史素材与生成结果。

职责包括：

- 保存聊天上传图片
- 保存聊天生成结果图
- 对前端提供历史素材列表、搜索、读取能力
- 支持聊天模式复用历史素材
- 支持批量编辑模式将素材库图片导入 PSD 素材池

### 4.3 PSD/PSB 工作流域

负责承接 `z_main_image` 的现有 PSD/PSB 能力。

职责包括：

- 上传模板文件 `PSD/PSB`
- 解析图层树
- 保存与读取替换组
- 导出与导入 Excel
- 上传商品素材
- 创建批量生成任务
- 提供任务状态、结果图片、下载压缩包、PSD 导出占位

## 5. 聊天生成域设计

### 5.1 main_image_banana 的整合方式

不采用“脚本子进程调用”或“单独 banana 服务”的方式。

建议做法：

- 将 `/home/sam/code/main_image_banana/playground/image_replace.py` 中的核心能力迁入主应用后端
- 把它整理成 `backend/app/services/chat/generation_service.py`
- 当前 `run_product_replace.py` 的流程改造成可被路由层直接调用的服务函数

### 5.2 迁入的核心能力

应迁入并重构的能力包括：

- 创建遮罩图
- 图片编码为 data URL
- 构造 chat-completions payload
- 读取 `.env` 中的接口配置
- 调用生图接口
- 从返回结果中提取图片 URL
- 下载返回图片
- 将结果图缩放到原始尺寸

### 5.3 聊天工作流 MVP

聊天生成请求的后端流程固定为：

1. 前端创建或进入一个会话
2. 用户上传图片或从素材库选择图片
3. 前端提交用户输入文本与附加图片列表
4. 后端读取该会话上一轮摘要 `last_summary`
5. 后端组装新的 rewrite prompt
6. 后端调用生图能力生成结果图
7. 后端把结果图保存到素材库
8. 后端写入 assistant 消息与本轮摘要
9. 接口返回本轮消息、结果图、摘要、可复用素材信息

### 5.4 rewrite prompt 策略

MVP 阶段只做基础版本。

后端每轮生成时组装以下信息：

- 上一轮摘要
- 当前用户输入
- 当前附加图片信息

摘要不做复杂压缩，只保留一段简洁文本，满足下一轮继续生成即可。

### 5.5 聊天生成结果处理

- 生成结果默认自动入素材库
- 同时关联到当前会话
- 这样可以支持：
  - 聊天内继续编辑
  - 后续从历史素材库复用
  - 批量模式中导入素材

## 6. 素材库域设计

### 6.1 定位

素材库是跨模式共享的基础设施，不是单独的业务流程。

### 6.2 素材来源

MVP 阶段支持以下来源：

- 聊天模式上传图片
- 聊天模式生成结果图
- 后续手动导入图片

### 6.3 数据职责

素材库需要保存：

- 基本文件信息
- 来源信息
- 所属会话
- 文件路径
- 可供前端直接访问的静态 URL

### 6.4 与 PSD 工作流的衔接

在批量模式下：

- 普通图片不能充当 `PSD/PSB` 模板
- 但可以作为商品素材导入 `psd_assets`
- 因此后端需要提供一个“从素材库导入到 PSD 素材池”的桥接接口

## 7. PSD/PSB 工作流域设计

### 7.1 z_main_image 的整合方式

不建议继续以“独立项目运行，再通过接口互调”的方式接入。

建议做法：

- 将 `/home/sam/code/z_main_image/backend/app` 下的现有能力迁入主应用后端
- 作为 `psd` 子模块存在
- 保持其服务边界与主要工作流不变
- 只做目录、表名、接口命名空间和配置路径整合

### 7.2 优先迁入的能力

优先迁入如下模块：

- `psd_service.py`
- `group_service.py`
- `excel_service.py`
- `asset_service.py`
- `job_service.py`
- 对应的 `routers`
- 对应测试

### 7.3 工作流保留原则

保留 `z_main_image` 当前主流程不变：

1. 上传 `PSD/PSB`
2. 解析图层
3. 创建替换组
4. 导出 Excel
5. 填写后导入 Excel
6. 上传商品素材
7. 创建批量生成任务
8. 下载结果图与导出结果

### 7.4 接口命名空间调整

为了避免和主应用其他接口混淆，统一挂到 `/api/psd/*` 下。

建议接口包括：

- `POST /api/psd/templates/upload`
- `GET /api/psd/templates/{template_id}/layers`
- `POST /api/psd/templates/{template_id}/groups`
- `GET /api/psd/templates/{template_id}/groups`
- `POST /api/psd/templates/{template_id}/groups/preview`
- `GET /api/psd/templates/{template_id}/excel/export`
- `POST /api/psd/templates/{template_id}/excel/import`
- `POST /api/psd/assets/upload`
- `POST /api/psd/assets/import-from-library`
- `POST /api/psd/jobs/generate/{template_id}`
- `GET /api/psd/jobs/{job_id}`
- `GET /api/psd/jobs/{job_id}/images`
- `GET /api/psd/jobs/{job_id}/download`
- `POST /api/psd/jobs/{job_id}/export-psd`

## 10. 配置设计

统一用 `.env` 配置。

建议至少包含：

- `IMAGE_API_BASE_URL`
- `IMAGE_API_KEY`
- `IMAGE_MODEL`

若 `main_image_banana` 当前变量名不同，迁入时统一改为主应用内部配置命名。

## 11. 实施顺序

### 阶段一：后端骨架

- 初始化 `backend/`
- 建立 `FastAPI` 入口
- 建立配置、数据库、文件目录初始化逻辑
- 建立公共健康检查接口

### 阶段二：素材库基础能力

- 建立 `library_assets` 表
- 完成素材上传、查询、静态访问
- 完成聊天上传图片入库能力

### 阶段三：聊天生成链路

- 迁入 `main_image_banana` 核心能力
- 整理为可复用 service
- 建立聊天会话、消息、摘要持久化
- 完成 `generate` 接口
- 完成结果图自动入素材库

### 阶段四：PSD/PSB 子模块迁入

- 迁入 `z_main_image` 后端核心服务与路由
- 调整表名、目录、配置与命名空间
- 保证模板解析、Excel、批量任务链路能继续运行

### 阶段五：素材库与 PSD 桥接

- 支持从素材库导入到 `psd_assets`
- 保证批量模式可以复用历史商品素材

## 12. 测试方案

### 12.1 聊天域测试

- 遮罩图生成正确
- payload 构造正确
- 接口返回图片 URL 提取正确
- 生成成功后结果图入库成功
- 多轮时能够读取上一轮摘要

### 12.2 素材库测试

- 上传素材成功
- 素材查询成功
- 搜索与排序正常
- 聊天结果图自动入库成功

### 12.3 PSD 域回归测试

迁入后继续保留并运行 `z_main_image` 现有关键测试，至少覆盖：

- 模板解析
- 图层树读取
- 替换组保存
- Excel 导入校验
- 批处理任务
- 结果输出

### 12.4 手工验收场景

至少验证以下端到端场景：

1. 聊天模式上传图片并生成结果图
2. 聊天模式第二轮继续编辑上一轮结果
3. 生成结果图能在素材库中看到
4. 批量模式上传 PSD/PSB 后进入模板工作流
5. 批量模式能从素材库导入商品素材并生成任务

## 13. 默认假设

本方案默认以下前提成立：

- 当前 MVP 为本机单用户使用
- 前后端都迁入当前主项目，不再把两个旧项目作为运行时依赖
- 聊天生成先只支持一条生成路径，不做多模型切换 UI
- 聊天接口暂不做流式返回
- PSD 工作流优先保持现有逻辑稳定，不额外引入新抽象

## 14. 后续扩展方向

当 MVP 跑通后，可继续扩展：

1. 支持多生图路径切换与技术路径对比
2. 对聊天摘要与 rewrite prompt 做更强的上下文压缩
3. 将素材库升级为更完整的数据库检索系统
4. 为 PSD 工作流补更高保真 PSD 导出能力
5. 增加多用户、权限、部署和监控能力

---

这份方案的核心思想是：

- **主应用统一承载前端与后端**
- **聊天生图迁入 banana 能力**
- **PSD/PSB 批量流程迁入 z_main_image 能力**
- **素材库作为全局共享基础设施**
- **先做单机可跑通，再谈复杂扩展**
